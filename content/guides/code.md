## 2026년 한국 개발자의 AI 코딩 풍경

채팅창에 코드 붙여넣고 묻던 시대는 끝났습니다. 지금은 AI가 **에디터·CLI·터미널 안에서 직접 파일을 편집하고 테스트를 돌립니다**. 한국 1인 개발자에게도 사이드프로젝트 진입장벽이 크게 낮아졌어요.

| 도구 | 특징 | 비용 | 적합 사용자 |
|---|---|---|---|
| **Claude Code (CLI)** | 터미널에서 직접 파일·테스트·git 조작 | Claude Pro/Max | CLI 친숙한 개발자 |
| **Cursor** | VSCode 포크, AI 통합 에디터 | 월 $20~ | GUI 선호, 풀스택 |
| **Windsurf** | Cursor 대안, agentic editing | 월 $15~ | Cursor 대안 |
| **GitHub Copilot** | 자동완성 중심 | 월 $10 | 가벼운 보조 |

> 한국 1인 개발자 가장 흔한 조합: **Claude Code + 본인 텍스트 에디터** (가성비), 또는 **Cursor 단일** (통합 경험).

## 핵심 패턴 — 작업별 프롬프트

### 1. 신규 기능 추가

```
다음 요구사항으로 [기능명] 을 [파일/위치]에 구현해줘.

요구사항:
- [구체적 동작 1]
- [구체적 동작 2]

제약:
- 기존 [패턴/스타일]과 일관성 유지
- [라이브러리·프레임워크] 사용
- 에러 케이스: [구체적 케이스]

테스트: [test framework]로 최소 3개 케이스 (정상 1 + 엣지 2)
```

**중요**: 제약(constraints)이 없으면 AI가 자기 마음대로 새 라이브러리 끌어옵니다. 기존 코드베이스 패턴을 한 줄로 명시하세요.

### 2. 버그 수정

```
다음 에러가 발생함:

[에러 메시지 전체 + 스택 트레이스]

관련 코드:
[file:line — 의심되는 부분]

원인 분석 후 최소한의 변경으로 수정해줘. 의도하지 않은 사이드 이펙트가 없도록.
```

**핵심**: "최소한의 변경" + "사이드 이펙트 없도록" — 안 그러면 AI가 주변 코드까지 리팩토링합니다.

### 3. 코드 리뷰

```
다음 PR을 시니어 코드 리뷰어로서 검토해줘.

[diff 또는 코드]

검토 항목:
1. 정확성 — 버그·엣지 케이스·보안
2. 가독성 — 네이밍·구조·주석
3. 성능 — 명백한 비효율
4. 관용적 스타일 — 이 언어·프레임워크 컨벤션

형식:
- 1줄 종합 판정 (Approve / Request Changes)
- file:line 형식으로 이슈 나열, 심각도(Critical / Important / Nit)
- 각 이슈: 무엇이 문제 + 왜 중요 + 수정 제안
- 마지막에 잘된 점 1~2개
```

### 4. 사이드프로젝트 0→1

```
다음 아이디어로 사이드프로젝트 MVP 구조를 잡아줘.

아이디어: [한 줄 설명]
타겟 사용자: [페르소나]
핵심 기능 3개: [기능1, 기능2, 기능3]
선호 스택: [Next.js + Supabase 등]

출력:
1. 디렉토리 구조 트리
2. 핵심 데이터 모델 (스키마)
3. 첫 1주일 작업 순서 (5~7개)
4. 잠재적 위험 / 의사결정 필요 지점
```

이 한 프롬프트가 사이드프로젝트 시작의 가장 큰 장벽을 낮춥니다.

## Claude Code 특화 — 한국 1인 개발자에 잘 맞는 이유

- **CLI에서 직접 동작** → IDE 락인 없음
- **MCP 도구 통합** → 외부 API·DB와 자연스럽게 연결
- **긴 컨텍스트 (200K+)** → 큰 코드베이스도 한 번에 이해
- **한국어 인스트럭션 안정적** → 한국어로 시켜도 결과 영문 자료와 거의 동급

## 자주 하는 실수

- **컨텍스트 없이 코드 한 토막만 줌**: 주변 파일·import·관련 함수도 함께 보여줘야 정확
- **테스트를 안 시킴**: "구현 + 테스트 + 둘 다 통과 확인"까지 한 번에 시키세요. 안 그러면 깨진 코드가 머지됨
- **AI에게 디자인 결정까지 맡김**: 아키텍처·기술 선택은 본인이. AI는 결정한 내용의 실행에 강합니다
- **변경 범위 제한 안 함**: "이 함수만 수정" 같이 명시 안 하면 주변 파일까지 손댐

## 보안 주의

- **시크릿 키·.env 노출 금지** — AI에 코드 붙여넣기 전 검색
- **데이터베이스 직접 조작 권한**은 신중히 줄 것 — AI가 의도치 않게 데이터 삭제할 수 있음
- **PR 머지 전 본인 검수 필수** — AI 코드를 그대로 main에 푸시하지 마세요

## 실전 워크플로 예시

본인이 한 사이드프로젝트 1주일 흐름:

| 일 | 작업 | AI 역할 |
|---|---|---|
| 월 | 아이디어 정리 + 스택 결정 | 브레인스토밍 + MVP 구조 잡기 |
| 화 | 프로젝트 스캐폴드 + 첫 페이지 | scaffold·기본 컴포넌트 작성 |
| 수 | 데이터 모델 + DB 연결 | 스키마·migration 작성 |
| 목 | 핵심 기능 1개 | 구현 + 테스트 |
| 금 | 핵심 기능 2개 | 구현 + 테스트 |
| 토 | 배포 + 도메인 연결 | 배포 스크립트·환경변수 |
| 일 | 본인이 수동 QA + 다듬기 | 버그 수정 보조 |

이 흐름으로 평균 **1주일 안에 사이드프로젝트 MVP**가 나옵니다. 사람 혼자였다면 2~3주 걸리던 작업.

## 마무리

AI 코딩은 코드 작성의 보조가 아니라 **개발 사이클 전체의 가속기**입니다. 다만 "AI가 다 알아서 한다"는 환상은 버리세요 — 아키텍처·보안·최종 검수는 사람의 책임입니다. 그 책임만 본인이 지면, 1인 개발의 생산성은 5년 전 5명 팀과 비슷해질 수 있습니다.

> 핵심: **제약 명시 + 테스트 동시 요구 + 보안 주의 + 본인이 최종 검수**.
